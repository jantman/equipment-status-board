# Tech-Spec: Slack UX Improvements

**Created:** 2026-05-10
**GitHub Issue:** [#34 Slack UX Improvements](https://github.com/jantman/equipment-status-board/issues/34)

## Overview

### Problem Statement

The Slack integration for ESB has four UX gaps that make it inconsistent and hard to use:

1. **Inconsistent emoji prefixes** in outbound notifications. `resolved` uses `:white_check_mark:`, safety-risk reports use `:warning:`, but other event types (new problem reports without safety risk, severity changes, ETA changes) have no leading emoji, so messages don't visually distinguish at a glance.
2. **`/esb-status` with no args is too shallow.** It shows only per-area counts (`X operational, Y degraded, Z down`) with no information about *what* is down/degraded. Users have to drill into individual equipment to learn anything actionable. There's also no hint about how to get more detail.
3. **`/esb-report` vs `/esb-repair` is confusing.** Both open create-record modals, but the difference (member-style report vs. technician-authored record) isn't clear from naming, and there's no Slack path for a technician to *act on* a report someone else filed.
4. **`/esb-update` requires a `repair-id` arg that isn't surfaced anywhere in Slack.** Technicians have no way to discover repair IDs without leaving Slack to look them up in the web UI, making the command effectively useless from inside Slack.

### Solution

A focused refactor of the Slack integration:

1. Add consistent leading emojis to every outbound notification event type and ephemeral status message, using a small fixed legend.
2. Extend `/esb-status` (no args) to list non-green equipment under each area with severity + brief description + ETA, and add a footer hint pointing at `/esb-status <area name>`. Extend `/esb-status <arg>` to match area name first (showing area-detail), falling back to existing equipment search.
3. Repurpose `/esb-repair` as the **technician dispatcher**: no args ŌåÆ modal listing open/active repair records, where the technician picks one and gets a follow-up modal with the high-frequency actions (Claim, Set ETA, Set Status, Resolve w/ Note). `/esb-report` keeps its current member-style problem-report flow. `/esb-update <id>` stays for in-depth multi-field edits ŌĆö same modal as today.

### Scope

**In Scope:**

- Add leading emojis to all outbound Slack notifications generated by `_format_slack_message()` in `esb/services/notification_service.py` (event types: `new_report`, `severity_changed`, `eta_updated`, `resolved`).
- **Add a new `status_changed` outbound notification event** so dispatcher status changes (and any other non-resolving status mutation) emit a Slack message. Without this, the dispatcher's "Set Status" / "Claim" actions would be silent on Slack ŌĆö re-creating the original Problem #1 inconsistency. Includes wiring in `repair_service.update_repair_record()`, a new `notify_status_changed` config key (default `'true'`), and a `_format_slack_message` branch with the legend's status-changed prefix.
- Add leading emojis to ephemeral confirmation messages in `esb/slack/handlers.py` (problem report submitted, repair created, repair updated, etc.) where appropriate / consistent with the legend.
- Extend `format_status_summary()` in `esb/slack/forms.py` to include per-area non-green equipment detail (severity, brief description, ETA) plus a footer hint for `/esb-status <area name>`.
- Extend `/esb-status <arg>` handler to first try to match an Area by name (case-insensitive). On match ŌåÆ render an area detail view (counts + non-green equipment with detail). On no area match ŌåÆ fall back to existing `equipment_service.search_equipment_by_name()` flow.
- Repurpose `/esb-repair` (no args) as the technician dispatcher: open a modal listing open repair records (id, equipment, area, status, severity, assignee). Technician selects one ŌåÆ second modal with action buttons or a compact form for: **Claim** (set assignee = self, status = Assigned), **Set ETA** (datepicker), **Set Status** (subset: In Progress, Closed - Duplicate, Closed - No Issue Found), **Resolve w/ Note** (status = Resolved + add timeline note).
- `/esb-repair <equipment-name>` keeps the existing technician-create-record flow. (A "Create new repair" button inside the dispatcher modal is deferred to a follow-up; out of scope here.)
- Keep `/esb-update <id>` as-is for full multi-field edits (status, severity, assignee, ETA, specialist description, note).
- Update Slack integration tests for all of the above.

**Out of Scope:**

- Redesigning Block Kit modals beyond what's needed for the new dispatcher and area-detail views.
- Changing the database schema, repair record state machine, or status-derivation logic.
- Adding outbound `assignee_changed` event. (`status_changed` IS in scope ŌĆö see In Scope ŌĆö to ensure dispatcher status changes are not silent.)
- Web-UI changes (this is purely a Slack-surface refactor).
- Translating commands to Slack global shortcuts or app home tabs.
- Renaming any of the three slash commands. (`/esb-update` may be downplayed in docs but is not removed.)
- Adding a separate `/esb-repairs` command (the listing lives at no-args `/esb-repair`).

## Context for Development

### Codebase Patterns

- **Slack handlers** live in `esb/slack/handlers.py` and are registered via `register_handlers(bolt_app, app)`. Every handler that touches the DB or services MUST wrap its body with `with _ensure_app_context(app):` ŌĆö see the docstring on `_ensure_app_context()` near the top of `esb/slack/handlers.py`. The helper is a no-op when an outer context already exists (tests run inside `app.app_context()` from the `app` fixture).
- **Line-number references in this spec are best-effort hints from spec-creation time.** Locate code by symbol/function name first; if a cited line range no longer matches, search for the function name and proceed.

- **Block Kit modal builders and message formatters** live in `esb/slack/forms.py`. View-submission `callback_id` strings tie modals to their submission handlers (current: `problem_report_submission`, `repair_create_submission`, `repair_update_submission`). New callbacks for this feature: `repair_dispatcher_submission` (Step A ŌåÆ push action modal) and `repair_action_submission` (Step B ŌåÆ execute action).

- **Service layer pattern** ŌĆö handlers and forms must NOT query models directly for business logic. Use:
  - `equipment_service.search_equipment_by_name(term)` ŌĆö partial, case-insensitive equipment search. Used by current `/esb-status <arg>`.
  - `equipment_service.list_areas()` ŌĆö non-archived areas ordered by name. Used by the new area-name resolver.
  - **NEW** `equipment_service.get_area_by_name(name)` ŌĆö to be added; case-insensitive exact name match against non-archived areas, returns `Area | None`.
  - `status_service.get_area_status_dashboard()` ŌĆö backs the no-args `/esb-status`. Returns `[{area, equipment: [{equipment, status: {color, label, issue_description, severity, eta}}]}]`.
  - `status_service.get_single_area_status_dashboard(area_id)` ŌĆö same shape for one area; raises `AreaNotFound` / `AreaArchived`. Used by the new `/esb-status <area-name>` path.
  - `status_service.get_equipment_status_detail(equipment_id)` ŌĆö adds `assignee_name` to status dict. Used by the existing single-equipment match path.
  - `repair_service.get_repair_queue(area_id=None, status=None)` ŌĆö backs the dispatcher list. Eager-loads `equipment.area` and `assignee` so we can build option labels without N+1.
  - `repair_service.update_repair_record(repair_record_id, updated_by, author_id, **changes)` ŌĆö handles `status`, `severity`, `assignee_id`, `eta`, `specialist_description`, `note`. The `note` kwarg appends a timeline entry in the same call. Used by every action in the dispatcher. Has a no-op guard: if the new value matches the existing value, no timeline entry is created (the source of AC 19a / AC 21a).
  - `repair_service.get_repair_record(repair_record_id)` ŌĆö fetches by id; raises `ValidationError` on not-found, but does NOT filter by status (closed records still return). Callers needing open-only records must filter on `record.status not in CLOSED_STATUSES` themselves (see Tasks 11, 12).
  - `repair_service.CLOSED_STATUSES` and `REPAIR_STATUSES` constants for status validation / option lists.

- **User resolution** ŌĆö `_resolve_esb_user(client, slack_user_id)` in handlers.py looks up the ESB User by Slack-profile email. Required for any technician-only command (gate on `esb_user.role in ('technician', 'staff')`). The dispatcher and action submission both need this for `assignee_id` / `author_id` / `updated_by`.

- **Outbound notifications** are queued (not sent inline) via `notification_service.queue_notification(notification_type='slack_message', target=channel, payload={'event_type': ..., ...})`. The worker calls `_format_slack_message(payload)` at delivery time (find both functions by symbol). **Emoji changes for outbound messages therefore live entirely in `_format_slack_message`** ŌĆö the queue payload shape stays the same. The new `status_changed` event also flows through this same queue.

- **Two-step modal flow with views_push.** When a view-submission handler wants to chain into a second modal instead of acknowledging done, it calls `ack(response_action='push', view={...})`. The pushed view inherits the user's modal stack so `Cancel` returns to the previous view. We use this for dispatcher ŌåÆ action transition. `private_metadata` carries `repair_record_id` from the dispatcher submission to the action submission.

- **Status emoji legend** already exists in `esb/slack/forms.py` as `_STATUS_EMOJI` (green/yellow/red). Extend by adding the prefix constants in `esb/services/notification_service.py` (since that's where outbound text is built) ŌĆö see Task 3 for the exact constant names and values.

- **Testing patterns:**
  - `tests/test_slack/test_handlers.py` ŌĆö every test uses `_register_and_capture(app)` at line 10 to capture handler functions from a `MagicMock` Bolt app, then invokes them as plain callables with mock `ack`, `client`, `body`, `view`. No live HTTP. New dispatcher tests follow this exact shape.
  - `tests/test_slack/test_forms.py` ŌĆö pure-function tests of formatters and modal builders.
  - `tests/test_services/test_notification_service.py::TestFormatSlackMessage` ŌĆö direct calls to `_format_slack_message` with payload dicts; `assert ':emoji:' in text` style. Existing four event-type tests need new emoji assertions added.
  - `tests/conftest.py` helpers `_create_area`, `_create_equipment`, `_create_user`, `_create_repair_record` build fixtures inline ŌĆö reuse them.

- **Slack Block Kit constraints to respect:**
  - Modal: max 100 blocks total.
  - `static_select`: max 100 options per element. `option_groups` lets you partition into named groups (max 100 groups), but the practical addressable option count is still bounded by Slack's element-level cap. **For this makerspace, total open-repair count is well under 100; dispatcher capacity is treated as bounded.** If ever exceeded, we'd add a search/filter ŌĆö explicitly out of scope here.
  - `option`/`option_group` `text`: max 75 chars (already truncated in `build_equipment_options`).
  - `option` `description`: max 75 chars (use for severity + assignee in dispatcher list).
  - `plain_text_input`: max 3000 chars; sufficient for notes.
  - `private_metadata`: max 3000 chars; we only need a single repair-id integer.

### Technical Decisions

- **Emoji legend (final, post-adversarial-review):**
  The legend has two surfaces. **Status indicators** (per-equipment, equipment-status color glyph) and **message prefixes** (event-type indicator at the start of an outbound or ephemeral message). `:warning:` is intentionally dual-use ŌĆö context disambiguates.
  - **Status indicators (in `_STATUS_EMOJI`, used inline beside an equipment item):**
    - `:white_check_mark:` ŌĆö green / Operational
    - `:warning:` ŌĆö yellow / Degraded
    - `:x:` ŌĆö red / Down
  - **Message prefixes (start-of-message glyph for outbound / ephemeral text):**
    - `:rotating_light:` ŌĆö new problem report (when not a safety risk)
    - `:warning: *SAFETY RISK* :warning:` ŌĆö safety-risk prefix on `new_report` and `severity_changed`. (Reuses `:warning:` deliberately; the bracketed `*SAFETY RISK*` text disambiguates from the status-indicator usage. Tests assert the full bracketed phrase, not bare `:warning:`.)
    - `:wrench:` ŌĆö severity changed (and ephemeral confirmation for dispatcher set_severity, if added later)
    - `:arrows_counterclockwise:` ŌĆö status changed (new outbound `status_changed` event AND dispatcher set_status / claim ephemeral confirmations)
    - `:calendar:` ŌĆö ETA set/changed (outbound + dispatcher set_eta confirmation)
    - `:white_check_mark:` ŌĆö resolved (outbound) AND generic success confirmations (problem reported, repair created, repair updated, repair resolved)
    - `:x:` ŌĆö error / not-found ephemeral messages (existing ŌĆö keep)
  - **Section headers (formatted blocks):**
    - `:bar_chart:` ŌĆö header for any status view that lists multiple equipment items (the existing summary header AND the new single-area-detail header). Reused intentionally; both surfaces are "status snapshot" pages.
    - `:mag:` ŌĆö search-not-found (existing ŌĆö keep)

- **`/esb-status <arg>` resolution order:** exact area-name match (case-insensitive) first; if no area matches, fall back to existing equipment search. The argument is stripped/lowered for matching. To support area lookup, add `equipment_service.get_area_by_name(name: str) -> Area | None` that filters non-archived areas by case-insensitive exact name. (Avoid partial area-name matching to keep the precedence rule unambiguous.) Document the resolution order in the Slack docs.

- **`/esb-status` (no args) detail format:** for each non-archived area, render the existing per-area count line, then under it list every non-green equipment item as a bullet with: severity emoji + name + brief issue description (truncated to ~80 chars) + ETA if present. Areas with all-green equipment still show their count line (zero down/zero degraded). End the message with a footer hint: `_Tip: try `/esb-status <area name>` for full details on one area._`

- **`/esb-status <area-name>` detail view:** uses `status_service.get_single_area_status_dashboard(area_id)` which already exists; render via a new `format_area_status_detail(area_data)` that lists every equipment item (green AND non-green) with status emoji + name + status label, and for non-green items appends issue description, ETA, assignee. This gives the "full details" promised by the footer hint.

- **`/esb-repair` (no args) dispatcher: two-step modal flow.** Step A: open a small "select repair" modal whose only block is a `static_select` listing open repair records. Step B: on submission, return `ack(response_action='push', view=<action_view>)` to push the action modal on top. The action modal is a single form with: a radio-button block ("Action": `claim` / `set_eta` / `set_status` / `resolve_with_note`), conditional helper inputs (an ETA datepicker visible always but only used when action=set_eta; a static_select for status with options *In Progress*, *Closed - Duplicate*, *Closed - No Issue Found* used only when action=set_status; a multiline note input used by `resolve_with_note` and optional everywhere else). Submission of the action modal dispatches to the right `repair_service` call based on the radio value. `private_metadata` carries `<repair_id>`.
  - Rationale: Slack does not support truly conditional input visibility in a static modal without `views_update` round-trips. A single action-form modal with one "Action" radio + a few generally-applicable optional fields is simpler and avoids the extra round-trip. Server-side, the submission handler reads the radio value and uses only the fields it needs.

- **Dispatcher option list capacity:** Slack `static_select` allows up to 100 options. For grouping, this design uses **option_groups grouped by area** so the user gets visual area separation. Capacity treated as bounded for this makerspace (per scope decision); pagination is out of scope. Open repair records are pulled via `repair_service.get_repair_queue()` (already eager-loads equipment + area + assignee). Option label format: `#<id> <equipment-name> ŌĆö <status>` (truncated to 75 chars per Slack limit). Description (`description` field on option, max 75 chars) carries severity + assignee for at-a-glance.

- **Action modal: claim semantics (post-F11).** "Claim" sets `assignee_id = <esb_user.id>` always. Status transition: ONLY when the record's current status is `New`, also set `status = Assigned`. For any other open status (`Assigned`, `In Progress`, `Parts Needed`, `Parts Ordered`, `Parts Received`, `Needs Specialist`), leave `status` untouched ŌĆö claim is purely an assignee swap. Closed records cannot be claimed (Task 11 closed-record guard rejects them at the dispatcher submission). Resolves the calling user via `_resolve_esb_user`.

- **Action modal: resolve_with_note semantics.** Sets `status = Resolved` AND appends the supplied note as a timeline entry (the existing `note` kwarg of `update_repair_record` already does this in one call). Note is required when this action is selected; validate at view-submission and return `ack(response_action='errors', errors={'note_block': 'Note is required when resolving'})` if missing.

- **`/esb-update` is preserved** as the "in-depth/detailed change" path. Same modal as today; the dispatcher does NOT replace it. (A "More optionsŌĆ”" button on the dispatcher's action modal is deferred to a follow-up ŌĆö out of scope here.)

- **Backward compatibility:** existing notification consumers (worker + queued payloads already on disk) keep working ŌĆö the only change is added emoji prefix in `_format_slack_message`'s returned text. No DB schema change, no payload shape change. Existing `tests/test_services/test_notification_service.py::TestFormatSlackMessage` tests will need to add emoji-prefix assertions for the previously emoji-less event types.

- **Ephemeral confirmation emojis (post-F18):** keep `:white_check_mark:` for generic success (problem-reported, repair-created via `/esb-repair <equipment>`, full-modal repair-updated via `/esb-update`) and `:x:` for errors. Dispatcher confirmations: `:arrows_counterclockwise: Claimed Repair #<id>: ...`, `:calendar: ETA set for Repair #<id>: ...`, `:arrows_counterclockwise: Status updated on Repair #<id>: ...`, `:white_check_mark: Resolved Repair #<id>` ŌĆö matches the legend's status-changed prefix for both claim and set_status.
